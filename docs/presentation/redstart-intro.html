Redstart

A language for authoring The Graph subgraphs.

AssemblyScript is a general-purpose language of questionable quality. Redstart is a DSL whose compiler knows it's building a Subgraph.

01 The need

Today a subgraph is three files held together by string matching.

A schema, a manifest, and AssemblyScript mappings: three loosely-coupled artifacts, stitched together by stringly-typed names and a manual graph codegen step.

schema.graphql
type Transfer @entity {
  id: Bytes!
  value: BigInt!
}
subgraph.yaml
eventHandlers:
  - event: Transfer(...)
    handler: handleTransfer
entities:
  - Transfer
mappings.ts
export function handleTransfer(
  e: TransferEvent
): void {
  let t = new Transfer(id)
}

graph build runs the AssemblyScript compiler, so type errors against the generated classes are caught. The rest isn't. Forget a .save() and the entity silently never persists. Let a handler's signature drift from the chain and it just never fires. Leave a required field unset and it fails at deploy, not at build. The failure you remember from AssemblyScript is rarely the compile error. It's the green build that dies three hours into a sync, on a field you'd stopped thinking about.

02 The language

AssemblyScript looks like TypeScript. It isn't.

It's a TypeScript-like subset compiled to WebAssembly, with its own compiler and garbage collector. The syntax is familiar, so you bring your TypeScript instincts. They turn out wrong in exactly the places that cost you a sync.

the nullability holedocumented
// total: a loaded entity, amount is a nullable field
let amt = total.amount   // amt: BigInt | null

amt          + ONE_BI    // ✓ compile error: nullable local, caught
total.amount + ONE_BI    // ✗ compiles clean: nullable property, missed

AssemblyScript narrows nullability for locals, but not for property accesses, so the two identical sums above compile differently. The Graph's own migration guide tells you to hand-write the null check, because the compiler won't.

  • Uninitialised reads compile, then crash. Touch a field that was never set and you get a clean build and a dead sync.
  • A bare-bones standard library. There's no Date type, so you hand-roll timestamp maths, and ordinary web3 libraries don't compile.
  • Semantics drift between releases. ** silently switched from float to integer results, variable shadowing was removed, and nullability changed from one version to the next.

The failures that matter compile clean and only surface at runtime. That's the problem.

03 Why a linter isn't enough

AssemblyScript can't catch a bug it doesn't understand.

It has no concept of an entity, a .save(), a derived field, or a reverted eth_call. So whole classes of subgraph bug are invisible to it. Redstart makes them unrepresentable: a compile error, or absent from the grammar entirely.

04 The graveyard

I've tried to leave AssemblyScript four times.

"I built Matchstick in 2021, the testing framework you run when you type graph test. I've been trying to get off AssemblyScript ever since." Building it meant learning AssemblyScript's memory model by heart, the exact monster that would eat the first escape attempt.

01
yogurt · fake it from outside. Pure Rust, emulating the AS memory model at the binary level. The output side worked, but reading event params back out of graph-node's AS-formatted memory was unfakeable from the guest side. Archived after 66 deploy iterations.
02
a native ABI · change graph-node (PR #6462). A parallel Rust ABI, ~1,450 lines, benchmarked at ~617k Transfer events/sec. The maintainers said no, and they were right. A second runtime is a permanent second surface. The maintenance burden was the product.
03
Graphite · become AssemblyScript. Reimplemented the AS runtime in Rust; shipped ERC-20 + ERC-721 live on Arbitrum One. But now I own a byte-for-byte runtime forgery, in lockstep with asc, forever.
04
liminal · leave subgraphs entirely. A WASIp2 component runtime, a third lane beside Subgraphs and Substreams. Real, but not a subgraph: no Proof-of-Indexing, no GraphQL on the network. Escape by emigration.

Every one fought the runtime, or fled it. The runtime was never the enemy.

05 The realization

Stop fighting the runtime.

  • ✕Don't fake it from outsideyogurt
  • ✕Don't patch the protocolnative ABI
  • ✕Don't reimplement the runtimeGraphite
  • ✕Don't flee the paradigmliminal

Replace the part you actually hate: writing it.

The AssemblyScript compiler is fine. It's battle-tested, maintained by people who aren't me, and the canonical path the whole network already trusts. What hurts is writing AS by hand, across three files that don't check each other.

graph-node doesn't load a language. It checks the manifest says wasm/assemblyscript, reads rtId / rtSize off each object's header, and matches host functions by name.

A memory layout, an allocator protocol, a string encoding: that's all it speaks. So let the official compiler emit it, and replace only the writing.

06 Library vs compiler

A library can't out-think the language it lives in.

graph-node doesn't run WASM. It runs AssemblyScript-shaped WASM. And graph-ts is just a library inside that language. Neither lets you escape what AssemblyScript can express. A compiler does.

Not all WASM is the same. Compile Rust, Go, or C to wasm32 and graph-node can't read a single entity out of it. It reads AssemblyScript's object headers (rtId / rtSize) and resolves host functions by exact name. The WASM target is fungible; the ABI is not. That coupling is why you can't just bring another language, and why the fix has to produce AssemblyScript, not replace it.
GRAPH-TS · A LIBRARY, INSIDE ASSEMBLYSCRIPT

Hands you types and bindings, then stops.

  • ✕The language's sharp edges are still yours: nullable values that compile silently then crash, uninitialised reads, no Date type.
  • ✕A forgotten .save() is still a silent bug that surfaces three hours into a sync.
  • ✕Codegen type-checks what you reference, but a drifted handler signature still never fires, and a missing required field still fails at deploy.
  • ✕Can't infer immutability, optimise the output, or prove any of it.
REDSTART · A COMPILER, ABOVE ASSEMBLYSCRIPT

Its own front-end. AssemblyScript is just the backend.

  • ✓Footguns are ungrammatical: no null, no silent nullable arithmetic, no unsaved entity.
  • ✓One checked program: schema ↔ manifest ↔ handlers, cross-validated.
  • ✓Whole-program analysis infers immutability and applies best-practice for you.
  • ✓Store-diffs the result byte-identical: near-perfect AS, and sure of it.

A library inherits AssemblyScript's limits. A compiler makes AssemblyScript an implementation detail.

07 What it does

One language. One source of truth.

Redstart unifies all three artifacts into one language, split across as many .red modules as you like, with mod/use just like Rust. It type-checks them against each other: entities can live in one module and the handlers that write them in another, and the compiler resolves and checks across all of them.

  • The event signature in the manifest is derived from the ABI by reference. Rename the event and it's a compile error.
  • There is no null: nullability is always explicit as Option<T>.
  • Entities are dirty-tracked and auto-saved, so forgetting .save() can't happen.
main.redone file
abi ERC20 from "./abis/ERC20.json"

entity Account {
  id: Id<Bytes>
  balance: BigInt
  label: Option<String>   // no null, ever
}

source Token {
  abi: ERC20
  network: mainnet
  address: 0xA0b86991c6218b36c1d19D4a2e9Eb0cE3606eB48
  startBlock: 6082465
}

handler on Token.Transfer(event) {
  let receiver = Account.loadOrCreate(
    event.params.to, { balance: BigInt.zero })
  receiver.balance = receiver.balance + event.params.value
  // auto-saved at handler end
}

08 How it does it

It compiles to the AssemblyScript you'd have written by hand.

transfer.redyou write
handler on Token.Transfer(event) {
  let sender = Account.loadOrCreate(
    event.params.from, { balance: BigInt.zero })
  let receiver = Account.loadOrCreate(
    event.params.to, { balance: BigInt.zero })

  sender.balance   = sender.balance   - event.params.value
  receiver.balance = receiver.balance + event.params.value
  // dirty-tracked, saved automatically
}
redstart build
mappings.tsit emits
export function handleTransfer(event: TransferEvent): void {
  let sender = Account.load(event.params.from)
  if (sender == null) {
    sender = new Account(event.params.from)
    sender.balance = BigInt.zero()
  }
  let receiver = Account.load(event.params.to)
  if (receiver == null) {
    receiver = new Account(event.params.to)
    receiver.balance = BigInt.zero()
  }
  sender.balance = sender.balance.minus(event.params.value)
  receiver.balance = receiver.balance.plus(event.params.value)

  sender.save()
  receiver.save()
}
loadOrCreate(…)→load() + null guard + init
No raw nullable deref survives; the absent case is handled by construction.
−  +→.minus()  .plus()
The explicit graph-ts form (bare +/− compile too). The real win is upstream: arithmetic on an Option is a compile error, where AS compiles nullable maths silently and crashes.
// auto-saved→.save() injected
Every dirtied entity is flushed at handler end, even across a helper's early return.
match { Ok / Err }→try_*() + !reverted
A reverted eth_call is forced to be handled, never an unguarded abort.

09 How it does it · the proof

The output is canonical. The store is provably identical.

The emitted AssemblyScript is readable and idiomatic, not an opaque intermediate. The official toolchain consumes it unmodified:

redstart build→ graph codegen→ graph build→ graph deploy
THE EJECT PATH

Abandoning Redstart costs nothing

If you ever walk away, you keep the generated AssemblyScript, and it keeps working with the standard toolchain. That defuses the bus-factor objection to betting production infra on a young language. The cost of leaving is zero.

THE KILL-GATE

A field-level store-diff

The whole bet is fidelity, so it's continuously falsified: deploy the generated subgraph beside an independently hand-written reference on a live graph-node, and diff the two stores at a fixed block. Identical, or the approach is wrong.

GATE GREEN
The ARB token on Arbitrum One produced a byte-identical store: 0 diffs across 10 Account + 13 Transfer entities at block 477,660,492. Indexing fidelity is no longer a hypothesis.

11 Why indexers care

The leaner subgraph, applied for you.

A human writes some AssemblyScript. Redstart writes every line of it, from a model that owns the schema, the manifest, and the mappings, so it applies the storage best-practices most subgraphs never get around to, uniformly and every time. It won't beat perfectly hand-tuned AS, but it's a floor no one forgets to raise.

Shipping today · automatic rewrites

immutable
Inferred immutability. The checker proves which entities are append-only (created, never mutated anywhere) and emits @entity(immutable: true) for you. No versioned block_range rows: up to ~19% faster indexing, ~48% less disk on Edge & Node's benchmark (workload-dependent).
prune: auto
Pruned by default. graph-node's own default is prune: never, which keeps the entire history for a bigger DB and slower queries. Redstart emits prune: auto, keeping only what serving needs.

Flagged today · warns, doesn't fix yet

The "stuck at 3%" lint (W020), an eth_call inside a loop with one serial blocking RPC per iteration, is caught before you deploy, alongside handler-shape lints (unfiltered block handlers, call-handlers on non-tracing chains). These warn. Turning them into automatic, gate-proven rewrites is next.
THE CLOSED LOOP
Because the store-diff gate is green (0 diffs on the arb-erc20 fixture), the emitted AssemblyScript can be optimised with a safety net: change the output, re-run the diff, ship only if the indexed store stays byte-identical. That's performance work that can't silently corrupt the data, which nothing else in the ecosystem can offer.

12 Looking ahead

Safer is nearly done. Faster is the frontier.

Redstart owns the codegen, so a performance optimisation is just a better lowering of the same .red, and the kill-gate proves every one byte-identical. Apply the rewrite, re-run the store-diff, ship only if it still reads 0 differences. Neither a language nor a library can pull this lever: speed that provably can't change your data. It's the roadmap's priority arc, with immutability inference and pruning already shipping.

FOR INDEXERS · FASTER SYNCS, LEANER STORAGE

The optimising compiler: every rewrite gate-proven.

  • Auto-fill startBlock from the contract's deployment block, so you stop scanning dead chains (the single biggest sync win).
  • Declare eth_calls for parallel, cached pre-fetch (~5-10×), or drop the call entirely when its result already rides in the triggering event.
  • Bytes ids atop inferred immutable, for the benchmark's full ~28% faster, ~48% less disk. Growing arrays rewritten to @derivedFrom, O(n²) → O(n).
  • Redundant loads coalesced and hoisted out of loops, down to one load + versioned write where there were many.
  • Hand-rolled rolling sums pushed into the DB as an @aggregation, so a write-contention hotspot goes away with zero handler code.
FOR DEVELOPERS · SAFER, AND EFFORTLESS

The language keeps closing gaps you never wanted.

  • The last footguns fall: entity-ID collisions, event-signature drift, and full determinism become compile errors.
  • redstart new --from 0x… scaffolds a whole subgraph from a contract address, in one command.
  • Agent-native: machine-readable diagnostics and an MCP server, so an AI can author and fix subgraphs in-loop.

Same network. Same byte-identical data. Just faster, safer, and finally pleasant to write.

13 Why Rust

Built in Rust: a platform, not a translator.

Redstart is a real compiler, written in Rust, with a full semantic model of your subgraph and the entire crates.io ecosystem behind it, none of which is ever coming to AssemblyScript. That's how it catches bugs, and it's also headroom: a compiler that understands your subgraph can answer questions about it no library or linter ever could.

Know a subgraph's cost before you index it. The network's biggest blind spot: nobody knows how large a subgraph will get, or how long it'll take to sync, until they're already deep into indexing it.

redstart check --costroadmap
  Transfer       immutable · ~90 B/row · 1 row / event
  Account        mutable   · ~120 B/row · ⚠ re-versioned every update
  Token.holders  [Account!]! · ⚠ unbounded, O(n²) on disk
  Mint handler    2 eth_calls, serial · ⚠ sync-bound
  ──────────────────────────────────────────
  ≈ 3 writes/event · storage super-linear · eth_call-bound

Redstart already builds the model this needs: every entity, every handler, every write. Turning it into a report an indexer reads before syncing is exactly what only a compiler can do.

  • The whole crates.io ecosystem. Redstart already runs on a real lexer (logos), rich diagnostics (miette), a proper language server (tower-lsp). Fuzzing, property tests and incremental analysis all follow, none of it coming to AssemblyScript.
  • One static binary. No Node, no Docker, no Postgres. The AssemblyScript toolchain needs all three.
  • Whole-program analysis at Rust speed: schema, manifest and handlers modelled together, fast enough to run in your editor as you type.

AssemblyScript is where a subgraph runs. Rust is where it finally gets understood.

14 Where it stands

Stage 0 is done. Early, but real and end-to-end.

Proven by porting a real subgraph, PaulieB14's Graph Horizon indexer with three contracts on Arbitrum One, and deploying it live to Subgraph Studio. It's scoped as a Graph-Foundation-grant public good in the lineage of Matchstick, the framework this whole story started with, not a venture bet. Its purpose was never to make subgraphs faster to write. It's to make staying on the decentralized network pleasant, after years of watching people reach for a centralized alternative the moment AssemblyScript bit them.
